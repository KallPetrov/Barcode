# 📱 Мобилно приложение за PDA терминал (Mobile Application Master Guide)

Този документ описва архитектурата, компилирането, хардуерната интеграция и режима на работа на мобилното Kotlin Android приложение на CALAC. То е създадено за използване на промишлени PDA терминали с вградени баркод скенери.

---

## 🏗️ 1. Компилиране, изграждане и управление на Product Flavors

Приложението е разработено на **Kotlin** с използване на модерната библиотека Jetpack Compose за изграждане на потребителския интерфейс.

### Системни изисквания за компилация:
- **Android Studio** (версия Hedgehog или по-нова).
- **JDK**: Java 17.
- **Android SDK**: API 34 (Android 14) като целева версия, с поддръжка назад до API 26 (Android 8.0).

### Gradle Product Flavors (Конфигуриране на средите):
За да се улесни управлението на адресите на API сървъра при смяна на средите, в `build.gradle.kts` са дефинирани продуктови разновидности (Flavors):
- **`dev` Flavor**:
  - Предназначен за локално тестване. Автоматично задава адрес на API сървъра `http://10.0.2.2:5000` (стандартен адрес за достъп до хост машината през Android емулатор).
- **`prod` Flavor**:
  - За реална складова работа. Насочва мобилния клиент към сигурен HTTPS адрес на реалния облачен сървър.

### Команди за компилация през терминал:
Навигирайте до `mobile/android/` и изпълнете:
- За компилиране на дебъг версия с Dev настройки:
  ```bash
  ./gradlew assembleDevDebug
  ```
- За компилиране на производствена версия:
  ```bash
  ./gradlew assembleProdRelease
  ```
Генерираният APK файл ще бъде записан в директория `app/build/outputs/apk/`.

## 🔌 2. Хардуерна интеграция чрез Android Intents

Приложението CALAC премахва необходимостта от ръчно въвеждане на символи чрез виртуална клавиатура. То комуникира директно с лазерния скенер на устройството чрез механизма на Android **Broadcast Intents**.

### Конфигуриране на профили за сканиране (Zebra & Honeywell):

1. **Zebra DataWedge профил**:
   - Създайте нов профил в DataWedge приложението на устройството (напр. име *"CALAC"*).
   - Свържете профила с пакета на приложението `com.calac.pda`.
   - Настройте **Intent Output**:
     - *Intent Delivery Method*: Broadcast Intent
     - *Intent Action*: `com.calac.pda.ACTION_BARCODE_SCANNED`
     - *Intent Category*: `android.intent.category.DEFAULT`
2. **Honeywell Scanner Edge профил**:
   - Настройте при успешна детекция на баркод устройството да излъчва Broadcast Intent със същия Action (`com.calac.pda.ACTION_BARCODE_SCANNED`).

### Обработка на данните в Kotlin кода:
Приложението съдържа `BarcodeReceiver` клас, който прихваща Intent-а, извлича сканирания низ и вида на разпознатия баркод (symbology) чрез следния формат:

```kotlin
val barcodeString = intent.getStringExtra("com.symbol.datawedge.data_string") ?: ""
val symbologyType = intent.getStringExtra("com.symbol.datawedge.label_type") ?: ""
```

Тези данни се опаковат в JSON формат чрез `com.google.gson.JsonObject` и се изпращат асинхронно към Backend API.

## 🗣️ 3. Гласово насочване и устни команди (Voice UI / VUI)

Една от ключовите иновации в CALAC е системата **Voice UI (VUI)**, която позволява на оператора да работи изцяло "със свободни ръце и очи" (hands-free, eyes-free).

### Гласов синтез (TTS - Text-to-Speech):
Класът `VoiceService` инициализира локалния Android TTS двигател на български език. При преминаване към нов ред от Picking задачата, асистентът прочита инструкциите на български език:
- *"Преминете на адрес Коридор 3, Рафт 1."*
- *"Пикирайте 4 броя от артикул Вода 500 милилитра."*

### Гласово разпознаване (STT - Speech-to-Text):
Приложението изисква разрешение `RECORD_AUDIO`. Класът `VoiceCommandService` слуша за гласови команди от оператора по време на работа. За разпознаване на думите се използва алгоритъм за сравнение на фонемно съвпадение на български език.

### Списък на поддържаните гласови команди:
- **"Синхронизирай"** — Стартира незабавна синхронизация на локалната база с API сървъра.
- **"Изход"** — Затваря текущия прозорец на задачата и се връща в главното меню.
- **"Повтори"** — Напомня на оператора текущата стъпка от Picking поръчката, прочитайки я отново.
- **"Къде съм?"** — Асистентът прочита адреса на складовата локация, която операторът е сканирал последно, за пространствена ориентация.

Операторът може напълно да деактивира VUI от менюто с настройки, което се запазва в локалните SharedPreferences на устройството.

## 🔄 4. Офлайн режим на работа и конфликтна резолюция

CALAC е проектирана като **Offline-first** мобилна система. Всички данни за складовите задачи и наличности се кешират в локална база данни SQLite чрез библиотеката **Room DB**.

### Офлайн Сценарий:
- Когато операторът влезе в зона на склада без Wi-Fi покритие, приложението не спира да работи.
- Всички сканирания на продукти, заприходявания и потвърждавания на поръчки се записват в локалната Room база и се маркират като "несинхронизирани".
- Заявките се подреждат в локална изпращаща опашка (Sync Queue).

### Алгоритъм за Синхронизация (Last-Write-Wins):
При възстановяване на интернет връзката, приложението автоматично или при изрична гласова команда стартира процес по синхронизация:
- **Сравнение на версии**: Мобилният Sync Engine извиква API `/api/sync/execute` и изпраща натрупаните локални промени.
- **Конфликтна резолюция**:
  - За да се предотвратят разминавания (напр. двама оператори едновременно променят броя продукти на един и същ рафт от различни устройства), системата използва стратегията **Last-Write-Wins (LWW)**.
  - Всеки запис в базата данни (Stock, Transfer Order line, Picking Stock line) притежава полетата `Version` (целочислен брояч на промените) и `UpdatedAt` (системно UTC време на последната промяна).
  - Сървърът сравнява времевите маркери и приема за коректен този запис, който е записан последен по време. По-старите данни се презаписват, а на мобилния телефон се връща актуализираната истинска версия на наличността.
- Този математически доказан модел гарантира 100% консистентност между всички PDA устройства в склада и централната база.
