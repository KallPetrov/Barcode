# 🔐 Сигурност, автентикация и роли (Security & Authentication Master Guide)

Този документ предоставя дълбок технически и експлоатационен поглед върху сигурността на платформата CALAC. Тук ще намерите описание на жизнения цикъл на JWT, защитните механични бариери, мулти-тенант изолацията на ниво база данни, както и изискванията за защита на личните данни (GDPR).

---

## 🔑 1. Двукомпонентна JWT автентикация и жизнени цикли

CALAC използва архитектурния модел на безсъстоятелна (stateless) автентикация с криптографски подписани JWT (JSON Web Tokens) токени, поддържани от дълготрайни ротиращи се `Refresh Tokens`.

### Жизнен цикъл на Access Token (JWT)
1. **Генериране**: При успешно изпращане на потребителско име и парола (или PIN) към `/api/auth/login` (или `/api/auth/pin`), сървърът генерира JWT токен, подписан чрез симетричен HMAC-SHA256 ключ.
2. **Payload Claims**: JWT съдържа следните криптирани метаданни (Claims):
   - `sub`: Уникален идентификатор на потребителя (`UserId`).
   - `unique_name`: Потребителско име.
   - `role`: Роля в системата (`Operator`, `Supervisor`, `Admin`).
   - `tenant_id`: Идентификатор на фирмата клиент, към която принадлежи потребителят.
   - `full_name`: Пълно име на потребителя.
   - `must_change_password`: Булев флаг за принудителна смяна на паролата.
3. **Продължителност на валидност**: По подразбиране 8 часа.
4. **Валидация**: Всяка входяща заявка се проверява от .NET Authentication Middleware на ниво HTTP заглавки (`Authorization: Bearer <token>`).

### Жизнен цикъл на Refresh Token (Token Rotation)
За да се избегне честа досадна идентификация на операторите по време на работа, се използват ротиращи се Refresh токени:
- **Съхранение**: Базата данни записва `RefreshTokens` в специална таблица, обвързана с `UserId`.
- **Ротация**: Когато уеб или мобилното приложение установи, че JWT е изтекъл, то прави POST заявка към `/api/auth/refresh`, изпращайки стария Refresh токен.
- **Двустранно анулиране**: Сървърът проверява валидността му, генерира изцяло нов Access Token и **изцяло нов Refresh Token**, като анулира стария. Ако прихванат стар Refresh токен се опита да се използва повторно, системата автоматично анулира всички сесии на този потребител, разпознавайки опит за пробив.

## 🚫 2. Защита с PIN код и блокиране при Brute-Force (PIN Lockout)

При използване на PDA устройства на терен, въвеждането на буквени пароли е бавно и неудобно. Затова складовите оператори влизат с 4-цифрен персонален PIN код.

### Алгоритъм на блокиране (Lockout Workflow)

Всеки път, когато оператор въведе своя PIN код в мобилното приложение, заявката преминава през защитна проверка в `PlatformServices.cs`:

```text
                  [ Въвеждане на PIN код ]
                             │
                             ▼
                 [ Валиден ли е PIN-ът? ]
                    /              \
               Да  /                \ Не
                  /                  \
                 ▼                    ▼
       [ Изчистване на ]     [ Увеличаване на брояча ]
       [ грешните опити ]    [   на грешни опити     ]
                 │                    │
                 ▼                    ▼
         [ Успешен вход ]     [ Достигнати ли са 5 опита? ]
                                      /              \
                                 Да  /                \ Не
                                    /                  \
                                   ▼                    ▼
                          [ Блокиране на акаунта ]  [ Връщане на грешка ]
                          [      за 15 минути    ]  [   "Невалиден PIN" ]
```

- **Параметри**:
  - Лимит на грешни опити: `5`.
  - Време за автоматично отключване: `15 минути`.
- **Логване**: Всеки неуспешен опит се записва в Immutable Audit Log, включително хардуерния код на устройството (`DeviceId`) и IP адреса.

## 🔄 3. Задължителна първоначална смяна на парола (Workflow)

След създаване на акаунт от складов мениджър или при системно нулиране на парола, потребителят се вписва в статус `MustChangePassword = true`.

### Пълен работен поток (Enforcement Workflow)

1. Потребителят се логва със служебната парола.
2. Сървърът издава JWT токен с вписан Claim `"must_change_password": "true"`.
3. При всяка следваща заявка, вграденият в API `PasswordChangeEnforcementMiddleware` прихваща заявката.
4. **Проверка по Бял Списък (Whitelist)**:
   - Middleware проверява пътя (Path) на текущата заявка.
   - Ако потребителят има активен статус за задължителна смяна на паролата, той има достъп **единствено** до следните три адреса:
     - `GET /api/auth/me` (за проверка дали сесията е все още валидна)
     - `POST /api/auth/change-password` (за въвеждане на новата парола)
     - `POST /api/auth/logout` (за изход от текущия профил)
5. **Прекъсване**: Всеки опит за достъп до `/api/items`, `/api/picking` или всеки друг складов ресурс извън белия списък се прекъсва моментално с HTTP статус `403 Forbidden` и детайлно съобщение.
6. **Промяна**: След като потребителят изпрати успешна POST заявка към `/api/auth/change-password` с нова, силна парола, флагът в базата се променя на `false`, JWT токенът се преиздава и пълният достъп до системата се възстановява автоматично.

## 🏢 4. Изолация на данните (Multi-tenancy Database Isolation)

CALAC използва споделен модел на база данни със строга логическа изолация за своите многобройни корпоративни клиенти (Tenants).

### Имплементация чрез Global Query Filters

Всички таблици, чиито данни принадлежат на конкретна фирма (напр. стоки, наличности, складови локации, поръчки), задължително внедряват интерфейса `ITenantEntity`:

```csharp
public interface ITenantEntity
{
    public Guid TenantId { get; set; }
}
```

При конфигуриране на базата данни в `AppDbContext.cs`, се настройват глобални филтри на заявките (Global Query Filters):

```csharp
modelBuilder.Entity<Item>().HasQueryFilter(x => x.TenantId == _tenantService.GetTenantId());
modelBuilder.Entity<Location>().HasQueryFilter(x => x.TenantId == _tenantService.GetTenantId());
modelBuilder.Entity<PickingOrder>().HasQueryFilter(x => x.TenantId == _tenantService.GetTenantId());
```

### Как работи в реално време:
- Когато уеб или мобилното приложение извика заявка, `TenantService` извлича `tenant_id` Claim-а от подписания JWT токен на потребителя.
- При изпълнение на SQL заявката, Entity Framework автоматично и невидимо вгражда клауза `WHERE TenantId = '...'` в SQL кода.
- По този начин, софтуерният код за четене на стоки изглежда просто като `_db.Items.ToListAsync()`, но е криптографски и логически невъзможно потребител от Фирма А да види или промени стоки на Фирма Б.

## 📋 5. Неизменим одит лог (Immutable Audit Log)

Всички оперативни действия, свързани със сигурността, промяната на наличности и изпълнението на поръчки, се записват в Immutable Audit Log.

- **Свойства на записа**:
  - `Id`: Уникален идентификатор (Guid).
  - `TenantId`: Уверен филтър за разпределяне по фирми.
  - `UserId` & `UserName`: Идентификация на извършителя.
  - `DeviceId` & `DeviceName`: Информация за използваното PDA устройство.
  - `Action`: Тип действие (напр. `LOGIN`, `STOCK_ADJUSTED`, `MANUAL_OVERRIDE`, `WAYBILL_GENERATED`).
  - `EntityType` & `EntityId`: Към кой обект се отнася промяната.
  - `Details`: Подробна JSON или текстова хронология на промяната.
  - `CreatedAt`: Точно време (UTC).
- **Неизменяемост (Immutability)**: API сървърът не предоставя методи за актуализиране (`UPDATE`) или изтриване (`DELETE`) на записи в таблица `AuditLogs`.

## 🛡️ 6. Криптиране на лични данни на купувача (GDPR / PII Encryption)

Платформата съдържа криптографски защити за лични данни (Personally Identifiable Information - PII). Данни на получатели при доставки (като Име, Телефон и Адрес за изпращане) се криптират на ниво база данни в таблица `Shipments`.

- **Технология**: Използва се `DataProtection` библиотеката в .NET 10 LTS.
- **Процес**:
  - Преди записване на нов Shipment в базата данни, стойностите на полетата `ReceiverName`, `ReceiverPhone` и `ReceiverAddress` се криптират чрез AES-256 симетричен ключ, управляван автоматично от операционната система.
  - При извикване на заявка от упълномощен оператор с подходяща роля, данните се декриптират в реално време преди изпращане на JSON отговора.
  - По този начин данните остават напълно нечетими при неоторизиран SQL дъмп на базата данни.
