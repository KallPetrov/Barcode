# ADR 001: Мулти-тенант архитектурен модел

## Статус
Прието

## Контекст
Системата CALAC трябва да поддържа множество независими клиенти (тенанти) в една инсталация. Има три основни подхода:
1. Database-per-tenant (Физическа изолация)
2. Schema-per-tenant (Логическа изолация)
3. Shared Database, Shared Schema (Изолация на ниво ред чрез `tenant_id`)

## Решение
Избран е вариантът **Shared Database, Shared Schema** с изолация чрез `tenant_id` колона и EF Core Global Query Filters.

## Обосновка
- **Лесна поддръжка**: Управлението на една база данни и една схема е значително по-лесно от гледна точка на миграции и инфраструктура.
- **Ниска цена**: Позволява обслужване на много малки тенанти с минимални ресурси.
- **EF Core Global Filters**: Нативното поддържане на глобални филтри в EF Core минимизира риска от "изтичане на данни" (data leak), като автоматично добавя `WHERE TenantId = ...` към всяка заявка.
- **ITenantEntity интерфейс**: Всички ентитита, които изискват изолация, имплементират този интерфейс, осигурявайки дисциплина в кода.

## Последици
- Всички SQL заявки (ако има нативни) трябва ръчно да включват `tenant_id`.
- Индексирането по `(tenant_id, id)` или `(tenant_id, code)` е задължително за производителност.
- Риск от "noisy neighbor" ефект, ако един тенант генерира прекомерен трафик (изисква Rate Limiting).
